# ESP32-S3 Radar OLED Hardware User Manual

## 1. 方案概述

本方案是一套基于 `ESP32-S3` 的超声波雷达距离显示终端，核心能力包括：

- 通过 UART1 与 Modbus RTU 雷达通信
- 将距离数据显示到 OLED 屏幕
- 通过 Wi-Fi 连接 OTA 固件服务器
- 定期上报设备在线状态、固件版本与测距信息
- 支持远程 OTA 升级

当前系统已经完成：

- OLED 显示方向修正
- 雷达 Modbus RTU 读取调通
- OTA 联合服务器调通
- 设备心跳与升级记录打通

## 2. 硬件组成

本硬件方案主要由以下模块组成：

1. ESP32-S3 主控板
2. OLED 显示屏
3. RS485 超声波雷达模块
4. Wi-Fi 网络
5. USB 串口/USB Serial JTAG 调试链路

说明：

- 你当前已经将原 RS485 信号侧转换为 TTL，并接入 ESP32-S3
- 因此 ESP32 侧现在直接通过 UART1 和雷达通信

## 3. GPIO 连接表

### 3.1 OLED 接口

OLED 使用 I2C：

| 功能 | ESP32-S3 GPIO | 说明 |
| --- | --- | --- |
| `SDA` | `GPIO41` | I2C 数据线 |
| `SCL` | `GPIO42` | I2C 时钟线 |

I2C 控制器：

- `I2C_NUM_0`

OLED 地址探测：

- 优先尝试 `0x3C`
- 次选 `0x3D`

### 3.2 雷达 UART 接口

雷达使用 Modbus RTU，通过 UART1 通信：

| 雷达侧 | ESP32-S3 GPIO | 说明 |
| --- | --- | --- |
| 雷达 `D (TX)` | `GPIO18` | 接 ESP32 `UART1 RX` |
| 雷达 `R (RX)` | `GPIO17` | 接 ESP32 `UART1 TX` |

UART 参数：

- 串口：`UART_NUM_1`
- 波特率：`115200`
- 数据位：`8`
- 校验位：`None`
- 停止位：`1`
- 流控：`None`

### 3.3 控制台接口

控制台日志当前通过 ESP32-S3 的 USB Serial JTAG 输出，启动日志中可见：

- `GPIO 44 and 43 are used as console UART I/O pins`

实际开发调试时，你当前使用的是：

- `COM5`

也就是你电脑上识别到的这块 ESP32-S3 调试串口设备。

## 5. Modbus RTU 通信说明

### 5.1 当前读取方式

系统使用标准 Modbus RTU 读保持寄存器方式获取距离值，报文结构如下：

请求帧：

- 从站地址：`0x01`
- 功能码：`0x03`
- 起始寄存器高字节：`0x00`
- 起始寄存器低字节：`0x10`
- 数量高字节：`0x00`
- 数量低字节：`0x01`
- CRC16：低字节在前，高字节在后

应答帧期望结构：

- 从站地址：`0x01`
- 功能码：`0x03`
- 字节数：`0x02`
- 数据高字节
- 数据低字节
- CRC16

### 5.2 校验逻辑

固件对返回帧做以下检查：

1. 响应长度必须为 `7` 字节
2. 从站地址必须匹配
3. 功能码必须为 `0x03`
4. 数据字节数必须为 `0x02`
5. CRC16 必须正确

任何一步失败，都会把本次轮询记为错误，并在状态区显示：

- `TIME`
- `CRC`
- `ERR`

如果成功，则显示：

- `OK`

## 6. OLED 显示说明

### 6.1 页面内容

屏幕显示分四行：

1. `RADAR WIFI` 或 `RADAR OFF`
2. `DIST xxxxx MM`
3. `MODBUS状态 OTA状态 IDxx`
4. `APP 固件版本`

例如：

- 第一行：显示 Wi-Fi 是否在线
- 第二行：显示毫米单位距离值
- 第三行：显示 Modbus 与 OTA 状态缩写
- 第四行：显示当前固件版本

### 6.2 显示方向

当前项目已经处理了 OLED 左右镜像问题。  
你之前看到“像从屏幕后面看”的现象，本质上是段映射方向不正确。

现在固件中采用的方向控制为：

- 扫描方向命令：`0xC0`
- 段映射命令：`0xA1`

配置项：

- `OLED_MIRROR_Y_AXIS = 1`

实际效果是：

- 屏幕进行水平方向镜像修正
- 文本显示恢复为正向可读

## 7. 内部任务调度逻辑

当前固件是一个多任务系统，核心任务如下。

### 7.1 渲染任务

任务名：

- `render_display`

职责：

- 每 `250 ms` 刷新一次 OLED 内容
- 从共享快照中读取 Wi-Fi、OTA、Modbus、距离、版本信息
- 整体重绘四行文本

优先级：

- `4`

### 7.2 雷达轮询任务

任务名：

- `radar_poll`

职责：

- 每 `500 ms` 轮询一次雷达
- 组帧 Modbus 请求
- 通过 UART1 发送请求
- 读取并校验返回数据
- 更新距离值、成功计数、失败计数
- 通过 USB 串口输出日志

优先级：

- `5`

这是当前业务最核心的实时任务之一，因此优先级比显示和心跳更高。

### 7.3 OTA 任务

任务名：

- `ota_task`

职责：

- 连接 Wi-Fi
- 获取 IP
- 拉取远端 manifest
- 比较版本
- 若有新版本则执行 OTA
- OTA 成功后重启

优先级：

- `5`

### 7.4 设备心跳任务

任务名：

- `device_heartbeat`

职责：

- 周期性向服务器上报设备状态
- 上传距离值、OTA 状态、版本号等

在 `v1.2.3-radar` 中的行为：

- 启动后先等待一个周期再上报
- 每 `60` 秒发一次
- Wi-Fi 未连接时不发
- OTA 正在进行时不发普通心跳

优先级：

- `2`

这个任务被故意放低，目的是不要抢占雷达轮询和 OTA 关键流程。

## 8. 状态快照机制

系统内部使用一个共享状态快照，里面包括：

- `wifi_connected`
- `ota_available`
- `ota_in_progress`
- `distance_mm`
- `slave_id`
- `poll_ok_count`
- `poll_error_count`
- `wifi_text`
- `ota_text`
- `modbus_text`
- `app_version`

这个快照通过互斥锁保护，供多个任务共享：

- 雷达轮询任务负责更新测距状态
- OTA 任务负责更新 OTA 状态
- 显示任务负责读取并绘制
- 心跳任务负责读取并上传

这种设计的优点：

- 任务之间职责分明
- 显示与上报都不直接访问底层硬件
- 状态统一，便于调试与扩展

## 9. v1.2.2 心跳卡顿问题说明

你观察到的问题非常关键：  
设备在心跳发生后，雷达轮询有时会暂停一段时间。

从调试结果看，主要原因是：

1. 心跳发得太频繁
2. 心跳 HTTP 超时过长
3. OTA 状态上报过于同步
4. 网络请求与 UART 轮询在时间上互相干扰

### 9.1 修复内容

`v1.2.3-radar` 已做以下修复：

- 心跳从 `15s` 改为 `60s`
- 心跳超时改为 `3s`
- OTA 中不再做大量即时上报
- 心跳任务优先级下降
- 首次心跳延后发送

### 9.2 实测结果

我已经在真实板子上完成了验证：

1. 设备从 `v1.2.2-radar` OTA 到 `v1.2.3-radar`
2. 升级成功并自动重启
3. 串口连续观察约 `80` 秒
4. 雷达 `500 ms` 轮询持续输出，没有再出现明显“卡住一段时间不读”的现象
5. OTA 服务器上的设备 `lastSeen` 也在新的分钟级周期正常更新

这说明当前修复方向是正确的。

## 10. 启动与工作时序

系统完整时序可以概括为：

1. 上电
2. Bootloader 读取分区与 `otadata`
3. 启动当前应用分区
4. 初始化 NVS、事件循环、网络栈
5. 初始化 OLED、UART、USB 调试接口
6. 启动显示任务
7. 启动雷达轮询任务
8. OTA 任务联网
9. 拉取 manifest
10. 无更新时继续正常测距
11. 有更新时下载新固件并重启
12. 新固件启动后重新进入上述流程

## 11. 使用建议

### 11.1 接线建议

- 确保雷达 TX 真的接到 ESP32 `RX`
- 确保雷达 RX 真的接到 ESP32 `TX`
- TTL 与 ESP32 必须共地
- OLED 供电、电平和 I2C 地址要与模块一致

### 11.2 调试建议

出现读不到雷达时，优先检查：

1. 波特率是否为 `115200`
2. 寄存器地址是否仍是 `0x0010`
3. 从站地址是否为 `1`
4. TTL 接线方向是否反了
5. 是否存在供电不稳
6. 是否有多个串口工具占用 `COM5`

### 11.3 版本升级建议

发布新版本时建议：

1. 修改 `PROJECT_VER`
2. 本地编译
3. 上传到 OTA 服务器
4. 检查 manifest 是否更新
5. 观察设备是否自动升级
6. 在设备管理页检查升级时间与版本变更

## 12. 总结

当前这套硬件方案已经形成完整闭环：

- `ESP32-S3` 负责控制、联网与 OTA
- `UART1` 负责雷达 Modbus RTU 通信
- `I2C` 负责 OLED 显示
- Wi-Fi 负责 OTA 与设备在线管理
- 服务器负责发布、记录与设备可视化

对于一个可维护、可远程升级、可持续迭代的雷达显示终端来说，这套设计已经具备较完整的工程形态。
